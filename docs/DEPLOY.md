# Деплой: бесплатно или недорого

Два варианта: **всё на Yandex Cloud** (одна ВМ) или **фронт на Vercel + бэкенд на Render** (бесплатные тиры с ограничениями).

---

## Сравнение

| | Yandex Cloud (одна ВМ) | Vercel + Render |
|--|------------------------|-----------------|
| **Стоимость** | Бесплатно по trial (~4000 ₽), далее ВМ ~1000–1500 ₽/мес | Бесплатно (лимиты) |
| **Сложность** | Средняя (SSH, Nginx, systemd) | Ниже (веб-интерфейс) |
| **Надёжность** | ВМ не «засыпает», диск постоянный | Render: сервис «засыпает» после 15 мин без запросов, пробуждение ~1 мин; БД на Render — 30 дней бесплатно, потом нужно продлевать/платить |
| **Фото обращений** | Хранятся на диске ВМ | На Render диск временный — при перезапуске файлы теряются (для продакшена нужен S3/внешнее хранилище) |

---

## Вариант 1: Yandex Cloud (одна ВМ)

Подходит, если нужен стабильный деплой без «засыпания» и с постоянным диском для фото.

- **Бесплатно:** при регистрации дают trial с кредитами (хватает на несколько месяцев малой ВМ).
- **Платно:** ВМ 2 vCPU, 2 ГБ RAM — ориентировочно 1000–1500 ₽/мес.

**Подробная пошаговая инструкция:** [DEPLOY_YANDEX_CLOUD.md](./DEPLOY_YANDEX_CLOUD.md).

Кратко: создаёте ВМ Ubuntu → ставите PostgreSQL, Python, Node, Nginx → клонируете проект → настраиваете `.env`, systemd, Nginx и при необходимости HTTPS (Certbot).

---

## Вариант 2: Vercel (фронт) + Render (бэкенд и БД)

Бесплатные тиры, но с ограничениями (см. таблицу выше).

### 2.1. Бэкенд и БД на Render

1. Зарегистрируйтесь на [render.com](https://render.com).

2. **PostgreSQL:**
   - Dashboard → **New** → **PostgreSQL**.
   - Create. Регион можно оставить по умолчанию.
   - После создания откройте базу: скопируйте **Internal Database URL** (или External, если будете подключаться не с Render). Формат: `postgres://user:password@host/dbname`.
   - Важно: бесплатная БД на Render живёт **30 дней**, потом её нужно продлить (платно) или создать новую и перенести данные.

3. **Web Service (FastAPI):**
   - **New** → **Web Service**.
   - Подключите репозиторий с проектом (GitHub/GitLab).
   - Настройки:
     - **Root Directory:** оставьте корень репозитория (где `main.py`, `requirements.txt`).
     - **Runtime:** Python 3.
     - **Build Command:**  
       `pip install -r requirements.txt`
     - **Start Command:**  
       `uvicorn main:app --host 0.0.0.0 --port $PORT`
   - **Environment:**
     - `PGHOST` — хост из URL БД (без порта, если 5432).
     - `PGPORT` — 5432 (или порт из URL).
     - `PGUSER` — пользователь из URL.
     - `PGPASSWORD` — пароль из URL.
     - `PGDATABASE` — имя БД из URL.
     - `ALLOWED_ORIGINS` — через запятую адреса фронта, например:  
       `https://ваш-проект.vercel.app`
   - Создайте сервис. Render выдаст URL вида `https://ваш-сервис.onrender.com`.

4. **Таблицы в БД:**  
   Схему (users, appeals, admin_categories и т.д.) нужно создать вручную или выполнить миграции. Таблица `appeal_photos` создаётся при первом запуске бэкенда (в коде есть `ensure_appeal_photos_table()`).

5. **Загрузки (фото):**  
   На бесплатном Render диск эфемерный — при перезапуске/редеплое файлы из `uploads/` пропадут. Для постоянного хранения нужен внешний объектный сторидж (S3, Yandex Object Storage и т.п.) и доработка кода загрузки.

### 2.2. Фронт на Vercel

1. Зарегистрируйтесь на [vercel.com](https://vercel.com), подключите GitHub/GitLab.

2. **New Project** → импортируйте тот же репозиторий.

3. Настройки сборки:
   - **Root Directory:** укажите папку `frontend` (или корень, если в настройках Build зададите путь к фронту).
   - **Framework Preset:** Vite.
   - **Build Command:** `npm run build` (если root — `cd frontend && npm run build`).
   - **Output Directory:** `dist` (для Vite в папке frontend — `frontend/dist`).
   - **Environment Variables:**
     - `VITE_API_URL` = `https://ваш-сервис.onrender.com`  
       В этом проекте бэкенд отдаёт API с корня (`/get_appeals`, `/new_appeal` и т.д.), поэтому указывайте только базовый URL сервиса без `/api`.

4. В коде фронта запросы идут на `VITE_API_URL` (в проде) и при необходимости к нему дописывается путь (например `/get_appeals`). Убедитесь, что бэкенд на Render отдаёт API по корню (например `/get_appeals`, `/new_appeal`), а не по префиксу `/api`. Если у вас на бэкенде всё под `/api`, тогда `VITE_API_URL` = `https://ваш-сервис.onrender.com/api`.

5. Deploy. Vercel выдаст URL вида `https://ваш-проект.vercel.app`.

6. В Render в переменной `ALLOWED_ORIGINS` добавьте этот URL фронта (через запятую, если несколько).

### 2.3. Проверка

- Откройте сайт на Vercel — должен грузиться интерфейс.
- Вход, регистрация, обращения — должны ходить на бэкенд на Render (сеть может резать CORS, если домен не добавлен в `ALLOWED_ORIGINS`).
- Учтите: после 15 минут без запросов бесплатный сервис на Render «засыпает», следующий запрос может выполняться с задержкой ~1 минуту.

## Что выбрать

- **Нужен стабильный и предсказуемый деплой, есть возможность платить за ВМ** → Yandex Cloud по [DEPLOY_YANDEX_CLOUD.md](./DEPLOY_YANDEX_CLOUD.md).
- **Нужен бесплатный вариант для теста/демо и допустимы ограничения (засыпание, 30 дней БД, временные загрузки)** → Vercel + Render по этой инструкции.

Для продакшена с фото и без «засыпания» обычно берут ВМ (Yandex Cloud или другой провайдер) или платный тариф Render + внешнее хранилище для файлов.
